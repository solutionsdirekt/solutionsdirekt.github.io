Conspicuous Collation Commits Case Crime!

Today we encountered an interesting bug. Turns out not only people can be (case) sensitive.

Peter Lamby
28 September 2020

We have a service pulling data from two different databases. Sometimes it would fail when putting the data together. Around one in 2000 entries could not be processed.

What the code did was this:

Map<String, Customer> customers = ... // filled by selecting some customers from table A.
                                      // The key is the customer email.

// I stripped some batching here
try (DatabaseServiceReleaser dbr = new DatabaseServiceReleaser(db2);
        Connection con = dbr.getConnection();
        // In the real code the ? are dynamically added since the size of `customers` varies.
        PreparedStatement stmt = con.prepareStatement("SELECT email FROM TableBInAnotherDatabase WHERE someFlag = 2 AND email IN (?, ?, ?)"))
{
    int paramPos = 1; // fun fact. JDBC starts counting with 1
    for (String email : customers.keySet()) {
        stmt.setString(paramPos++, email);
    }
    try (ResultSet rs = stmt.executeQuery()) {
        while (rs.next()) {
            customers.get(rs.getString(1)).setSomeFlag(true);
        }
    }

3 out of ~8000 customers caused this code to fail with a NullPointerException. Can you guess line and cause?

I'll give you a tipp.

The exception occours on line 16.

This suprised me. I looked again at the code and was sure this was not possible. We had a list of emails and passed them to the DB. And now the emails somehow did not exist in our map anymore. Debugging finally helped me understand the problem when I saw the problematic email.

The email was test@example.com in TableA but Test@example.com in TableB. And sure enough. Looking at the collation of the database revealed Latin1_General_CI_AS. The CI means case insensitive.

The fix was simple. We now lowercase the emails in our map and also when retrieving the associated customer again.

Looking back the bug was not really hidden. I can think of a few reasons why I did not figure out the problem without looking and the failing email address.

  1. The setup is kinda uncommon. Usually you don't pull two strings from separate databases which are supposed to be equal. Normalisation should make sure you only have one copy of that string.

  2. Normally you would compare the strings on the sql server directly instead of reading them into memory first. The error would not have occoured there.

  3. The application swallowed the exception. So some time was needed to debug and reproduce the issue. It took a while to find a customer entry which revealed the bug.
